01

클로드와 클로드 코드의 차이

Q. 클로드(Claude)랑 클로드 코드(Claude Code)는 뭐가 다른가요?

한 문장으로: 클로드는 "대화하는 AI", 클로드 코드는 "내 컴퓨터에서 일하는 AI"입니다.

구분Claude (클로드)Claude Code (클로드 코드)
주 용도채팅·질문·글쓰기파일 편집·자동화·스킬 실행
내 컴퓨터 파일 접근✗ (업로드해야 봄)✓ (직접 읽고 쓰기)
터미널/명령 실행✗✓
사용 장소웹, 모바일 앱, 데스크톱 앱VS Code, 터미널, 웹·앱(Code 전용)

도식으로 보면 이런 관계예요.

              Anthropic (회사)
                    │
               Claude (모델)
                    │
        ┌───────────┼───────────┐
        │           │           │
      Claude      Claude      Claude
      (앱/웹)      Desktop     Code
        │         (+Cowork)      │
      대화 중심    파일 작업     내 컴퓨터
                 (샌드박스)     직접 제어

둘 다 같은 Claude 모델을 쓰지만, 어디까지 손이 닿는지가 달라요. 클로드는 대화창 안에서 작업하고, 클로드 코드는 내 컴퓨터 속 파일을 읽고 수정하는 코드를 짭니다.

02

로컬 파일을 분석할 때 쓰는 기능: Skill과 Agent

Q. 내 컴퓨터의 파일(엑셀, 이미지, 노트)을 클로드 코드로 분석하려면 뭐가 필요한가요?

두 가지 기능을 알면 됩니다. Skill과 Agent예요.

  • Skill = 레시피. 정해진 순서대로 실행되는 명령 모음. /hello-world처럼 / 슬래시로 부름. 한 번 만들어두면 같은 작업을 반복할 수 있음. 특정 작업을 하고나서 '스킬로 저장해줘'라고 말하면 스킬이 만들어짐.
  • Agent (Subagent) = 전문가 역할을 맡은 작은 AI. "탐색 전문", "리뷰 전문", "계획 전문" 같은 역할을 줘서 큰 작업을 나눠 시킬 때 좋음.

언제 뭘 쓰나?

  • 매번 같은 순서로 해야 하는 일 → Skill
  • 한 번에 여러 파일을 훑거나 큰 맥락을 파악해야 하는 일 → Agent

예: "매주 월요일에 지난주 노트 백업하기"는 Skill. "프로젝트 전체 파일을 읽고 스타일 가이드 위반을 찾아줘"는 Agent.

03

함께 알면 좋은 것: MCP, API

Q. MCP랑 API는 뭐예요?

둘 다 "클로드가 바깥 서비스와 연결되는 방법"입니다.

  • API (Application Programming Interface): 두 프로그램이 서로 말을 주고받는 규격. 예를 들어 구글 캘린더 API로 일정을 가져오거나, 슬랙 API로 메시지를 보냄. "이 주소에 이런 형식으로 요청하면 이런 답이 돌아온다"는 약속.
  • MCP (Model Context Protocol): 앤트로픽이 만든 "AI 전용 연결 규격". API보다 한 층 위에서 동작하며, 클로드가 슬랙·피그마·구글 드라이브·노션 같은 서비스와 대화하듯 주고받게 해줌. MCP 서버 하나만 연결하면 해당 서비스의 여러 기능을 자연어로 부를 수 있음.

한 줄 정리: API는 프로그램 사이의 통신 규격, MCP는 그 API를 AI가 쓰기 편하게 포장한 것.

몰라도 알아서 진행됩니다.

05

VS Code와 터미널

Q. VS Code, 터미널이라는 단어가 낯설어요

  • VS Code (Visual Studio Code): 마이크로소프트가 만든 무료 에디터. 개발자들이 제일 많이 쓰는 프로그램 중 하나. 메모장처럼 글을 쓸 수도 있고, 여러 파일을 동시에 열어볼 수 있고, 클로드 코드 같은 확장 프로그램을 붙여서 기능을 키울 수 있음.
  • 터미널: 컴퓨터에게 문장(명령)으로 지시하는 검은 창. 맥에서는 "터미널.app", 윈도우에서는 "PowerShell" 또는 "명령 프롬프트". 예를 들어 ls라고 치면 지금 폴더의 파일 목록이 나옴. 마우스 없이 키보드로 움직이는 방식.

둘 다 처음엔 익숙하지 않지만, 쓰다 보면 "내가 컴퓨터한테 직접 말을 걸고 있네"라는 느낌이 듭니다.

06

클로드 코드를 VS Code에서 쓰면 좋은 이유

Q. 왜 VS Code에서 쓰는 걸 제일 추천하나요?

한 화면에서 네 가지가 동시에 보이기 때문이에요.

  1. 파일 트리: 내 프로젝트의 폴더 구조가 왼쪽에 펼쳐져 있어서 "어떤 파일을 건드리는지" 한눈에 보임
  2. 에디터: 클로드가 바꾼 내용이 그 자리에서 바로 하이라이트됨 (빨간색 = 지운 것, 초록색 = 추가한 것)
  3. 클로드 채팅창: 오른쪽에 대화창. 파일을 드래그해서 "이 파일 봐줘" 식으로 바로 전달 가능
  4. 터미널: 아래쪽에 붙여놓고 필요할 때 바로 명령어 실행

Cowork나 모바일은 쉽지만 "내가 지금 뭐가 바뀌는지" 확인하기 어려워요. VS Code는 모든 변화가 실시간으로 보입니다. 여러 일의 진행 상황을 점검하며 진행하기에 가장 좋은 환경이에요.

08

문제가 생겼을 때 들이면 좋은 습관

Q. 에러가 나거나 막히면 어디서 물어봐야 해요?

무조건 클로드 코드 화면에서 물어보세요. 구글, ChatGPT, 슬랙으로 먼저 가지 마세요.

이유 3가지:

  1. 클로드 코드는 지금 열려 있는 파일과 폴더 상태를 이미 알고 있음. 같은 질문이라도 "내 맥락 안에서" 답해줌.
  2. 에러 메시지를 그대로 붙여넣으면 클로드가 직접 파일을 읽고 원인을 찾아줌.
  3. 대화 기록이 남아서 다음에 같은 문제가 날 때 "지난번에 이거 어떻게 풀었었지?"라고 바로 물을 수 있음.

'이 에러 원인이 뭐지.'라고 바로 물어보세요.

09

여러 컴퓨터에서 같은 작업을 이어가려면: iCloud, Git

Q. 회사 컴퓨터랑 집 컴퓨터에서 번갈아 쓰고 싶은데 어떻게 해요?

두 가지 방법이 있습니다.

  • iCloud Drive (또는 구글 드라이브, 드롭박스): 작업 폴더를 클라우드 동기화 폴더 안에 두면 자동으로 두 컴퓨터에 똑같이 유지됨. 저장만 하면 끝. 설정할 것이 거의 없음.
  • Git (+ GitHub): 작업 폴더를 저장소로 만들어 변경사항을 커밋하고, 다른 컴퓨터에서 pull로 받아옴. 수동이지만 누가 언제 뭘 바꿨는지 기록이 남고, 되돌리기 쉬움.
10

iCloud vs Git, 뭘 쓸까

Q. 그럼 iCloud랑 Git 중에 뭘 써야 해요?

공통점

  • 여러 컴퓨터에서 같은 파일에 접근 가능
  • 원본이 한 곳에만 있지 않아서 "컴퓨터가 고장나도 파일은 살아 있음"

차이점

항목iCloudGit
동기화 방식자동·실시간수동 (commit → push → pull)
기록최신 상태만 보관변경 이력 전부 보관
되돌리기하루 단위 정도어느 시점으로든 되돌리기 가능
협업파일 공유 링크브랜치·머지로 여럿이 작업
진입장벽거의 없음명령어 몇 개 익혀야 함
민감 파일실수로 다 올라감.gitignore로 제외 가능

추천

  • 혼자 쓰고 기록은 필요 없음 → iCloud
  • 기록·되돌리기·협업이 필요함 → Git
  • 둘 다 하고 싶음 → iCloud 동기화 폴더 안에 Git 저장소를 두는 것은 피하세요. 충돌이 자주 생깁니다. GitHub 같은 원격 저장소를 따로 두는 것이 안전합니다.
11

내 노트를 클로드 코드와 연결하기: 에버노트, 노션, 옵시디언

Q. 그동안 쌓아둔 노트를 클로드 코드가 읽게 하려면 어떻게 해요?

노트 서비스마다 연결 방법이 달라요.

서비스저장 위치클로드 코드 연결추천도
옵시디언내 컴퓨터 안 마크다운 파일폴더 경로만 알려주면 바로 읽고 쓰기 가능★★★
노션노션 서버Notion MCP 또는 Notion API 필요 (설정 있음)★★
에버노트에버노트 서버API는 있지만 활용이 어려움. 수동 export 추천★

공통점: 어떤 서비스든 결국 "텍스트를 저장하는 곳". 클로드 코드는 그 텍스트에 닿을 수 있는 길만 있으면 읽고 쓸 수 있음.

차이점: 파일이 내 컴퓨터에 있느냐, 서버에 있느냐. 옵시디언처럼 내 컴퓨터에 있으면 연결이 끝난 것과 같음. 노션·에버노트처럼 서버에 있으면 "가져오는 다리"가 한 번 더 필요함.

옵시디언을 쓰고 있다면 이미 최상의 조건을 갖춘 것이고, 노션을 쓰고 있다면 Notion MCP 설치부터, 에버노트를 쓰고 있다면 "이참에 옵시디언으로 옮겨볼까"를 한 번 고려해보는 것도 방법입니다.

12

명령하거나 스킬 만들 때 들이면 좋은 습관 6가지

Q. 실수를 줄이려면 어떤 습관을 들이면 좋을까요?

  1. 보안: API 키, 비밀번호, 토큰은 .env 파일에만. 절대 스킬 파일이나 채팅창에 적지 말 것. 깃에 올릴 때는 .gitignore에 .env를 꼭 추가.(git 쓰는 경우)
  2. 백업: 중요 작업 전에는 Git 커밋 한 번. 클로드가 파일을 크게 바꾸는 작업을 시작할 때는 "바꾸기 전에 먼저 커밋 한 번 해줘"가 안전장치.(git 쓰는 경우), 백업 문서를 만들어 달라는 습관을 들여야 함.
  3. 테스트: 자동화하기 전에 "먼저 한 번만 실행해서 결과를 보여줘"로 시작. 결과가 맞으면 이후에 반복 실행 추가.
  4. 지침: CLAUDE.md 같은 지침 파일을 프로젝트 폴더에 두면 클로드가 매번 먼저 읽음. 하지 말아야 할 것, 쓰는 용어, 톤앤매너를 여기에 적어두면 반복 설명이 사라짐.
  5. 토큰 아끼기(1): 명확한 경로 지정. 필요한 파일만 열어주기. 경로를 정확하게 지정해주기. 어떤 스킬을 진행하기 전에 어떤 경로를 참고하는지 먼저 공유하게 하기. 클로드가 엉뚱한 곳을 뒤지는 일이 확 줄어듭니다.
  6. 토큰 아끼기(2): /compact로 불필요한 대화 정리. 대화가 길어져서 컨텍스트가 무거워지면 /compact를 한 번 쳐주세요. 지금까지 주고받은 내용을 요약본으로 압축해서 이어갑니다.
13

제일 중요한 원칙

Q. 클로드 코드를 오래 잘 쓰려면 어떤 마음가짐이 필요한가요?

네 가지를 기억해주세요.

① 남의 스킬을 바로 쓰지 말고, 맥락을 파악해보세요

책을 사는 것만으로 내 지식이 되지 않듯, 타인의 스킬을 얻게 됐다면 사용해보기 전에 .md 파일을 열어 차근차근 읽어보세요. 어떻게 짜여 있는지, 왜 이 순서인지 보는 것만으로도 "아, 이렇게 짜면 되는구나" 이해됩니다. 같은 명령이라도 내 업무 상황과 남의 업무 상황은 다릅니다. "이 사람은 왜 이걸 이렇게 만들었을까?"를 한 번 되짚으면, 내 상황에 맞게 고쳐 쓸 수 있어요. 다른 사람이 어떻게 일하는지 업무 방식이 보이는거죠. 새 스킬을 떠올리는 아이디어의 원천이 됩니다.

③ 정리에서 멈추지 말고 액션플랜과 트리거까지

정리를 한데서 만족감을 느끼고 끝내면 그냥 정리만 한 것일 뿐입니다. "이걸 매주 월요일 아침에 자동으로 해주고, 결과를 DM으로 보내줘"까지 가야 자동화예요. 모든 스킬은 트리거(언제)와 결과(어디로), 액션 플랜⭐️이 있어야 생산성을 높일 수 있습니다. So what에 대한 답까지 만드세요.

④ 기록이 많을수록 결과물의 품질이 올라갑니다

대형 언어 모델(LLM)이 좋은 결과물을 내려면 읽을 수 있는 자료와 맥락이 많아야 합니다. 내 가이드라인, 지침, 과거 작업 기록, 회고, 후기, 메모. 많으면 많을수록 클로드가 나처럼 일할 수 있어요.

역설적이지만, AI를 잘 쓰려면 사람이 기록을 더 많이 해야 합니다. AI가 대신 기록해주는 게 아니라, 내가 남긴 기록 위에서 AI가 일하는 거예요. 가이드, 지침, 후기, 회고, 일지. 만들 수 있는 건 다 만들어두는 습관이 가장 큰 차이를 만듭니다.
