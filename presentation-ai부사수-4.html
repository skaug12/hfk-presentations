News 1 · Claude Opus 4.7

앤트로픽 — "한 번에 책 한 권을 읽고, 사진의 픽셀까지 본다"

2026년 4월 공개된 새 모델 Claude Opus 4.7. 한 번에 다룰 수 있는 맥락이 1M 토큰(≈책 한 권 분량)으로 늘고, 이미지도 3.75MP 고해상도까지 바로 읽습니다.

무엇이 달라졌나

  • 1M 토큰 맥락 — 녹취록 + 자료 + 템플릿을 한 번에 읽히고 작업
  • 2576px / 3.75MP 고해상도 이미지 — 기존 1568px / 1.15MP에서 대폭 상향
  • 이미지 좌표 1:1 매핑 — 스크린샷 위 요소 위치를 픽셀 단위로 지목 가능

Claude Code에서 바로 활용

모델을 바꾸는 건 /model 한 줄. 같은 스킬이라도 무거운 작업은 4.7, 단순 변환은 Sonnet/Haiku로 배치하면 됩니다.

"맥락이 길어진다"는 게 왜 중요한가

지금까지는 녹취록이 길면 요약본을 먼저 만들어 넣어야 했습니다. 4.7은 녹취록 원문 + 템플릿 + CLAUDE.md 규칙을 통째로 넣고 한 번에 리뷰 초안을 만들어냅니다.

HFK 기준 체감 포인트

  • 후기 생성 — 녹취 원문을 그대로 읽고 발언을 놓치지 않음
  • 시즌 레터 — 이전 6호치를 전부 참고해 톤 일치
  • 이미지 분류 — 카드뉴스 레이아웃의 여백/정렬까지 판단

한 줄 요약 — "원문을 잘라서 넣을 필요가 없어졌다."

News 1 · 비개발자에게 의미

"요약부터 맡기던 시대가 끝난다"

전에는 이랬습니다

  • 녹취 60분짜리를 먼저 요약해서 넣기
  • 템플릿 + 레퍼런스 후기 몇 편만 골라서 넣기
  • 이미지는 작게 줄여서 넣기

"자르는 것" 자체가 시간이고, 자르는 과정에서 맥락이 빠짐.

지금은 이렇게 됩니다

  • 녹취 원문 + 이전 후기 10편 + 템플릿을 통째로
  • AI가 직접 "중요한 발언이 뭔지" 판단
  • 카드뉴스 원본 PNG를 그대로 읽고 조정 제안

두 가지 시사점

  1. "요약"이 사람의 일이 아니게 됨 — 원문을 통째로 넘기고, AI에게 관점을 지정하는 게 더 정확한 결과를 줌.
  2. "이미지를 잘 설명하는 능력"이 의미를 잃음 — AI가 픽셀을 직접 본다는 것은, 디자인 피드백도 말로 풀 필요가 없어진다는 뜻.

우리에게 주는 신호 — "AI가 다루기 좋게 잘라주는 일"이 줄어드는 중. 오히려 우리가 해야 할 건 관점을 명확히 주는 것.

News 2 · Claude Code × Figma

디자인 도구와 AI가 왕복하는 시대

피그마가 Figma MCP 서버를 공개하면서, 클로드 코드가 피그마 파일을 직접 읽고, 반대로 클로드 코드로 만든 UI를 피그마 프레임으로 보내는 왕복 흐름이 가능해졌습니다.

두 방향 모두 가능

  • Figma → Code — 피그마 프레임 URL을 붙이면 작동하는 React/HTML 코드 생성
  • Code → Figma — VS Code에서 만든 UI를 피그마 프레임으로 내보내기
  • 디자이너는 수정 가능한 피그마 파일로, 개발자는 돌아가는 코드로 받음

MCP가 뭔지 한 줄로

AI가 외부 도구(피그마·슬랙·구글)를 통일된 방식으로 조작하게 해주는 규격. 도구마다 따로 연동할 필요가 줄어듭니다.

HFK 맥락에서

  • 카드뉴스 HTML → 피그마 프레임으로 바로 옮기기 (export-to-figma)
  • 피그마 시안 → 클로드가 읽고 Imweb HTML로 변환
  • 대시보드 시안을 피그마에서 잡고 클로드가 구현

3회차에 소개한 DESIGN.md와 합쳐지면, "우리 브랜드 규칙 + 피그마 원본 + AI 구현"의 한 덩어리가 됩니다.

핵심 변화 — 디자인이 "이미지"에서 "AI가 읽고 쓰는 파일"로 바뀌는 중.

News 2 · 의미

왜 디자이너·비개발자에게 의미 있을까

"전달"이 사라짐

예전엔 디자이너가 피그마를 만들고 개발자에게 "전달"했음. 이제는 AI가 중간 번역을 하니 왕복이 실시간으로 일어남.

디자인이 버저닝 가능해짐

코드와 피그마가 하나의 원본처럼 움직이면, "어느 시안이 최신인지" 문제가 줄어듦. git에 올려두고 관리 가능.

"Skill"이 확장됨

1회차에 이야기한 Skill이 이제 디자인 도구까지 호출함. "카드뉴스 만들어줘" → 바로 피그마 파일까지.

교훈 — "AI로 뭘 만든다"가 텍스트/코드를 넘어 디자인 결과물까지 뻗어가는 중. 1인 운영자가 디자이너 역할까지 겸할 수 있는 폭이 넓어집니다.

News 3 · Claude Skills

"Skill"이 공식 기능이 되었다

앤트로픽이 Skill을 Claude의 공식 기능으로 정식화. 우리가 .claude/commands/에 쌓아온 스킬들도 같은 개념의 표준화된 형태로 올라갑니다.

Skill이란

  • 같은 작업을 정해진 순서로 반복 실행하게 만든 "레시피"
  • 마크다운 한 파일로 정의. 코드 아님
  • 스킬 안에서 다른 스킬을 호출, MCP 도구 호출 가능

3회차와의 연결

3회차에서 소개한 카드뉴스 자동화 가이드도 Skill이었습니다. 이제 "Skill을 만든다"는 개념이 공식화되면서, 스킬 공유/설치도 쉬워지는 중.

HFK는 이미 52개

  • 44개 → 52개로 늘어남 (2주간 +8개)
  • 시즌 운영, 콘텐츠 생성, 슬랙 자동화, 대시보드 관리 등
  • 운영자 1명이 커뮤니티 운영의 대부분을 스킬 호출로 처리

스킬마다 적합한 모델

Opus 4.7(긴 맥락·판단), Sonnet 4.6(일반 판단·실행), Haiku 4.5(단순 변환). 스킬 52개를 3등급으로 재배치한 이야기는 뒤(Output 1)에서 자세히.

※ 세부 기능은 Anthropic 공식 문서 기준이며, 실제 활용 사례는 HFK 운영 환경 기반입니다.

evaluate-notes 스킬과 연동

기존 /evaluate-notes 스킬이 이 인덱스를 참조. 시즌 오픈 2주 전부터 일일 체크 가능.

왜 필요했나

시즌 오픈 직전엔 "무엇이 부족한지"를 감으로만 판단해왔음. 파트너 소개가 빠졌거나, 일정이 아직 미확정이거나, 커리큘럼이 옛날 버전이거나.

지표를 세워두면 "어떤 상품이 먼저 보완되어야 하는지"가 자동으로 잡힘.

의미 — 시즌 오픈 전 Panic 대신 체크리스트로 운영. 다음 시즌에도 그대로 재사용.

Output 3 · 다중 컴퓨터 동기화 v2

두 대의 컴퓨터에서 같은 작업을 끊김 없이

3회차 Output 4의 후속편

  • 3회차: "새 컴퓨터로 옮기기 — 두 파일이면 끝"
  • 4회차: "두 대에서 동시에 쓸 때 꼬이지 않게"

환경 이관에서 상시 동기화로 한 단계 진화.

핵심 구조 3가지

  • memory 폴더 symlink + git 관리 — 두 대가 같은 feedback을 즉시 공유
  • run-daily-note Step 0 원격 동기화 선행 — 작업 시작 전 자동 pull
  • docs/setup-new-computer.md 가이드 — 재현 가능한 셋업 순서

왜 필요했나

두 대를 오가며 일하기 시작하면서 "어느 쪽에 어떤 변경이 있었는지"를 매번 확인해야 하는 상황이 생김. 특히 feedback 메모리는 한쪽에만 반영되면 다음날 같은 실수를 반복하는 문제가 있었음.

원칙 — 컴퓨터가 아니라 git이 진실의 원본. 두 대는 둘 다 체크아웃된 워크트리일 뿐.

Output 4 · 멤버 성장 경로 매핑

통합 멤버십 시트 → UUID 기반 멤버 마스터

build_members_master.py 신규

  • 통합 멤버십 시트를 읽기 전용으로 가져옴 (시트는 절대 수정 안 함)
  • 이메일 · 전화번호 · 이름으로 동일인 식별
  • 멤버 단위 UUID 부여 & 영속 매핑 (재실행해도 같은 UUID)
  • 출력: members.json · members_index.json · orphan_rows.json

시즌 운영 대시보드에 반영

  • 재등록 멤버 수, 재등록율 목표(72%)
  • 첫등록 / 재등록 구분 집계
  • 팀 체인 추적 가능한 기반 완성

"성장 경로"란

한 멤버가 여러 시즌·여러 팀을 거치며 쌓는 궤적. 예: 25봄 리더십첫줄 → 25여름 AI부사수 → 26봄 강점차별화.

기존엔 시트가 시즌별로 분산되어 있어 한 사람의 궤적을 추적하기 어려웠음. UUID 기반 마스터가 있으면 "이 멤버에게 다음 시즌 뭘 추천할까"가 데이터로 답 가능.

의미 — 멤버 한 명을 시즌의 합이 아니라 커리어의 궤적으로 보는 전환.

시간 회수

왜 T인가? S가 "하던 일을 더 잘"이었다면, T는 "안 해봤던 일에 발을 담그는 것". AI 덕분에 진입장벽이 크게 낮아진 영역입니다.

T in Detail

T = Try — 평소 못한다고 생각했던 것

"전공이 아니라서", "툴을 몰라서", "배운 적 없어서" — 그동안 손대지 못했던 일을 AI를 옆에 두고 한번 해보는 영역.

T 영역의 특징

  • 결과물이 남음 — S와 달리 습관이 아니라 산출물 중심
  • 첫 시도는 무조건 엉성함 — 완벽을 목표로 하면 시작 못 함
  • AI에게 "처음 해봅니다"를 알려주는 게 핵심 — 맥락이 바뀜
  • 한 번 뚫으면 가능 영역이 크게 넓어짐 — 전문 분야의 진입 문턱 낮춤

슬기의 T 영역 사례 (실제 시도한 것)

  • /edit-premiere-xml — 영상 편집 잘 못함 → 프리미어 프로 XML을 텍스트로 편집
  • /update-card-news — 디자인 비전공 → 카드뉴스 v3 (a/b/c)까지 반복 개선
  • Apps Script — 프로그래밍 처음 → 구글폼 자동 알림, 팀추천위젯
  • /ga-report — 데이터 분석 약함 → GA4 월별 페이지 리포트 자동
  • Growth Bot — Slack 봇 개발 처음 → 9종 스킬을 봇으로 노출

공통점: 전부 "이건 내 분야가 아니야"에서 시작.

T in Detail

T 영역 스킬은 어떻게 뚫는가

예시 1 — 카드뉴스 디자인 (비전공자)

처음 명령: "인스타 카드뉴스 5장 만들어줘"

  1. v1 — 엉성함. 폰트 어색, 여백 이상
  2. 클로드에게 "뭐가 어색한지" 찍어서 알려줌
  3. v2 — 나아짐. 하지만 정렬이 틀림
  4. Figma로 열어 직접 정렬 수정 → 스킬에 규칙으로 반영
  5. v3 (a/b/c) — 세 가지 스타일로 분화됨

예시 2 — 영상 XML 편집 (비전공자)

프리미어 프로 시퀀스를 XML로 내보낸 후, 클로드가 텍스트로 편집하고 다시 임포트. 타임라인 UI를 전혀 안 써도 됨.

T 영역 공통 구조

  1. 맥락 고백 — "이 분야 처음입니다. 기본 개념부터 알려주세요."
  2. 작은 목표 — "완벽한 결과" 대신 "일단 한 번 돌아가는 것"
  3. 관찰 — 첫 결과의 어디가 어색한지 구체적으로 지목
  4. 반복 — "이것만 고쳐보자"를 3~5번

기억할 것
T 영역의 가장 큰 장벽은 기술이 아니라 "난 못할 것 같다"는 감정. AI에게 "처음입니다"를 말하는 순간 문이 열립니다.

T in Detail · 예시 3

웹캠 모션 인식으로 인터랙티브 아트 만들기

왜 이 예시가 T의 정수인가

  • "미디어 아트는 예술+공학 전공자가 하는 것"이라는 인식
  • 실제로는 HTML 한 줄 + ml5.js PoseNet이면 시작 가능
  • 그동안 진입 자체가 막혀 있던 영역 — AI로 문이 열림

결과물이 "감각적"이라 성공 경험이 오래 남음.

처음 명령 (맥락 고백)

# 맥락 고백
코딩 처음. HTML 띄우는 법만 앎.
# 작은 목표
웹캠으로 내 손 위치를 인식해서
화면에 원이 따라다니게 하기.
# 원하는 결과 형식
HTML 파일 1개, 브라우저에서 바로 실행.
# 체크 방법
코드 전에 뭘 설치해야 하는지 먼저 알려줘.

진화 과정 (5번의 반복)

  1. v1 — p5.js + ml5.js 연결. 콘솔에 손 좌표만 찍힘
  2. v2 — 좌표 위치에 원 그리기. "아, 되는구나"
  3. v3 — 원에 색 그라데이션. "어떤 색 좋을까?" 선택지 3개 요청
  4. v4 — 이전 위치 잔상 남기기. 물감 번지는 느낌
  5. v5 — 손 속도에 따라 붓 두께 변화. "내가 만든 도구"가 됨

실제로 본인이 알아야 한 것

  • HTML 파일을 브라우저로 열 줄 아는 것
  • "카메라 권한을 허용"을 누를 줄 아는 것
  • "이게 왜 이래?"를 클로드에게 물어볼 수 있는 것

p5.js도 PoseNet도 수학도 몰라도 됨.

T의 정수 — "나는 못해"에서 "이걸 내가 만들었다"까지, 한 세션 2시간이면 충분한 시대.

Workshop · Step 1

STEP 1 — 못한다고 생각했던 것 떠올리기 (5분)

"못한다"의 정의

  • "내 전공이 아니라서" 미뤄둔 것
  • "툴을 배운 적이 없어서" 시도 못 한 것
  • "전문가한테 맡겨야 하는 일" 같아 포기한 것
  • "나는 저런 건 못해"라고 습관적으로 말한 것

업무도 좋고, 취미도 좋음 (디자인·영상·코딩·데이터 분석 등).

예시 (참고용)

  • 간단한 홍보 포스터를 직접 만들어보기
  • 엑셀 매크로로 반복 작업 자동화
  • 유튜브 영상 자르기 / 자막 넣기
  • 웹캠 모션 인식으로 인터랙티브 아트 만들기 (p5.js + 카메라)
  • 간단한 랜딩페이지 만들기
  • 설문 데이터 시각화 (차트 그리기)
  • 블로그/뉴스레터 레이아웃 꾸미기

"미디어 아트는 전공자만 한다"는 편견이 AI로 깨지는 중.

활동 — 노트에 3개를 적어보세요. "못할 것 같은 이유"도 한 줄씩.

Workshop · Step 2

STEP 2 — 가장 가벼운 1개 고르기 (5분)

고르는 기준 3가지

  1. 실패해도 남에게 피해 안 가는 것 — 회사 공식 결과물 말고 내 실험물
  2. 결과물이 한 장/한 파일로 나오는 것 — 범위가 좁을수록 좋음
  3. 검색만으로는 배우기 벅찼던 것 — 옆에서 알려줄 사람이 필요했던 영역

하지 말 것

  • "한 달 걸리는 프로젝트"를 T에 넣지 마세요
  • "전문가 수준으로 마무리" 목표로 잡지 마세요
  • 여러 명이 같이 봐야 하는 결과물은 보류

활동 — 3개 중 가장 작고 무해한 것에 동그라미 하나. 오늘 우리가 만들 T 도전의 주제입니다.

Workshop · Step 3

STEP 3 — 첫 명령 만들기 (10분)

T 영역 4칸 양식

  1. 맥락 고백 — "이 분야 처음입니다. 기본부터 알려주세요."
  2. 작은 목표 — "일단 한 번 돌아가는 결과 1개"
  3. 원하는 결과 형식 — "HTML 파일 / PNG 1장 / 차트 1개"
  4. 체크 방법 — "결과 보여주고 물어봐주세요"

S와의 차이: "처음 해봅니다"를 명시하는 것.

완성 예시 — "홍보 포스터 처음 만들기"

# 맥락 고백
디자인 비전공. 포스터 만들어본 적 없음.
# 작은 목표
인스타용 정사각형 1장이면 충분.
# 원하는 결과 형식
HTML 파일로 먼저 보여줘. 문구 3줄, 배경색 1개.
# 체크 방법
완성 전 시안을 브라우저로 미리 보여줘.

활동 — 노트에 4칸을 그리고 각각 한 줄씩 채워보세요. "처음입니다"를 꼭 쓸 것.

Workshop · Step 4

STEP 4 — 실행 + 3번 피드백 (15분)

실행 방법

  1. VS Code 클로드 코드 채팅창 열기
  2. 4칸 내용을 그대로 한국어로 풀어서 입력
  3. 첫 결과를 받음 — "엉성할 것을 예상"
  4. 어디가 어색한지 구체적으로 1~2개 피드백
  5. 3번째 반복까지만 진행. 그 이상은 나중에

자주 막히는 곳

  • "잘 모르겠어요" → 클로드에게 선택지 3개를 제시해달라고 요청
  • 결과가 전혀 엉뚱함 → 맥락 고백을 더 자세히 (왜 처음인지, 어디서 막혔는지)
  • 한 번에 완벽하게 하려는 유혹 → "다음 1개만 고쳐줘"로 좁히기

입력 예시 (그대로 써도 됨)

> 디자인 비전공이고, 포스터를
  직접 만들어본 적 없어.
  인스타용 정사각 1장이면 충분해.
  먼저 HTML로 시안을 만들어줘.
  문구는 '4월 독서 모임 모집' 한 줄.
  시안 보고 피드백할게.

> (결과 받은 뒤)
  배경색이 너무 진해.
  글자는 가운데 정렬로.

3번째 피드백까지만. 그 이후엔 멈추고 기록하세요.

Workshop · Wrap

오늘 뚫은 T를 어떻게 키워갈까

오늘 밤

  • 첫 결과물을 파일로 저장
  • "어색했던 것 → 고친 것" 한 줄 메모
  • "처음엔 못할 줄 알았지?" 한마디

이번 주

  • 같은 주제로 결과물 2개 더 만들기
  • 처음과 3번째 결과 비교
  • 슬랙에 시도 공유 (자랑해도 OK)

다음 시즌

  • 반복해서 하던 T를 스킬로 저장
  • "이제 할 수 있는 것 리스트"에 추가
  • 다른 T 영역으로 확장

기억하세요 — T 영역 성공의 지표는 "결과물의 퀄리티"가 아니라 "내가 할 수 있는 일의 리스트가 늘었는지"입니다.
